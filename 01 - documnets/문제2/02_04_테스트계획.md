# 02_04 · 테스트 계획 – MediGo

> 문제 2 · 품질 보장 전략

## 1. 테스트 범위
- 포함: 핵심 사용자 플로우(처방전 업로드 → 약국 선택 → 주문 생성 → 배달 현황 확인)
- 제외: 외부 결제, 실시간 GPS 연동 등 비범위 기능

## 2. 테스트 종류
- 단위(Unit): DB 모델, 유틸 함수
- 통합(Integration): API + DB, API + 프론트 연동
- E2E: 실제 UI 클릭 플로우(수동 데모 스크립트 중심)

## 3. 전략 및 커버리지 목표
- 최소: 핵심 플로우 happy path 100% 수동 검증  
  - (1) 알림→앱 진입→약국 검색→주문 생성→배송 완료  
  - (2) 재고 없음 약국 선택 시 오류 메시지 확인  
  - (3) 네트워크 끊김 시 fallback 메시지 확인  
- 선택: pytest로 API 레벨 테스트 작성(시간 허용 시)

## 4. 테스트 케이스 예시
| ID | 시나리오 | 단계 | 기대 결과 | 비고 |
|----|----------|------|-----------|------|
| TC-01 | 약국 검색 | “서울시청” 입력 후 검색 | 리스트와 지도에 약국 표시 | |
| TC-02 | 재고 없음 약국 클릭 | available=false 약국 선택 | 모달에 “재고 없음” 메시지 | |
| TC-03 | 주문 생성 | 배달 가능한 약국에서 주문 버튼 클릭 | `/api/orders` 200 응답 + ID 발급 | |
| TC-04 | 배달 현황 표시 | 주문 직후 배달 화면 이동 | 라이더 아이콘 이동 시작, ETA 표시 | |
| TC-05 | 처방전 업로드 | 예시 이미지를 업로드 | 서버에 저장되고 업로드 성공 메시지 | 옵션 |

## 5. 수용 기준(AC)
- API 응답 p95 < 500ms (로컬 기준)
- 오류 시 사용자에게 의미 있는 메시지 표시
- 크리티컬 버그(플로우 단절) 0건
